SEC 01

なぜセンサフュージョンが必要か

加速度センサと地磁気センサは、それぞれ単独では姿勢角を完全には決定できない。 お互いが苦手とする成分を補い合うのがセンサフュージョンの出発点。

1.1加速度センサだけでは yaw が分からない

静止・等速に近い状態では、加速度センサは重力加速度ベクトル g をほぼそのまま検出する。 機体が傾けば、機体座標系で見た g の向きが変わるので、そこから roll と pitch は計算できる。 しかし、機体を鉛直軸まわりに回転（＝yaw が変化）させても、重力ベクトルは常に真下を向いたままなので、 機体座標系で見た g の向きは変化しない。つまり yaw は原理的に加速度センサからは求まらない。

1.2地磁気センサは yaw が分かるが、傾くと誤差が出る

地磁気センサは、地球磁場ベクトル（水平成分＋伏角による垂直成分）を機体座標系の3軸で測定する。 機体が水平であれば、水平面内の磁場成分から方位（yaw）をそのまま計算できる。 しかし機体が傾くと、世界座標系では磁場ベクトルは変わらなくても、 機体固定の X / Y / Z 軸で見た磁場成分の内訳が変わってしまう。 この状態で「水平前提」の式にそのまま代入すると、方位が大きくずれる。これが傾斜補正（tilt compensation）が必要な理由。

POINT

まとめると：加速度センサは傾き（roll/pitch）だけが分かり yaw は分からない。地磁気センサは方位（yaw）の情報を持つが、 傾いた状態のままでは正しく取り出せない。だから「加速度＋ジャイロで roll/pitch を求め、それを使って磁力計データを水平化してから yaw を出す」という 2段階のフュージョンが必要になる。

SEC 02

相補フィルタによる roll / pitch 推定

加速度センサとジャイロの弱点を打ち消し合わせて roll / pitch を推定する、最も基本的なフュージョン手法。

2.1加速度とジャイロ、それぞれの弱点

加速度センサから atan2 で計算する角度は、振動や機体の並進加速度が乗ると 短期的にノイズだらけになる。一方で長期的には（重力方向という物理的な基準があるため）ドリフトしない。
ジャイロは角速度を積分して角度を得るため、短期的には滑らかで正確だが、 わずかなバイアス誤差やノイズが積分によって蓄積し、長期的にはドリフトしていく。

2.2相補フィルタのブロック図

相補フィルタは「ジャイロ積分（高周波成分を信頼）」と「加速度計から求めた角度（低周波・長期成分を信頼）」を 重み α で混ぜ合わせる、シンプルな1次フィルタ。

FORMULA — この機体の軸配置（up = X, forward = Z）で書き直した式
COMPLEMENTARY FILTER — roll / pitch
roll_accel  = atan2(accelY, accelX)
pitch_accel = atan2(-accelZ, sqrt(accelY^2 + accelX^2))

roll  = α(roll  + gyroX*dt) + (1-α)*roll_accel
pitch = α(pitch + gyroZ*dt) + (1-α)*pitch_accel   ← α = 0.95〜0.98
注意

一般的なIMUチュートリアルでは pitch はふつう gyroY と対になるが、 本ロケットは軸配置が X=上・Z=前・Y=横 という特殊な割り当てのため、上式のとおり roll には gyroX、pitch には gyroZ を使う。他資料の式をそのままコピーしないこと。

IMPLEMENTATION — Arduino / C++（Sensor.cpp の変数名に合わせる）
// 相補フィルタで roll / pitch を更新する
// _accelX/_accelY/_accelZ [g], _gyroX/_gyroZ [deg/s], dt [s]
const float ALPHA = 0.96f;

float _roll  = 0.0f;
float _pitch = 0.0f;

void updateRollPitch(float dt) {
  float rollAccel  = atan2(_accelY, _accelX);
  float pitchAccel = atan2(-_accelZ, sqrt(_accelY * _accelY + _accelX * _accelX));

  _roll  = ALPHA * (_roll  + _gyroX * dt) + (1.0f - ALPHA) * rollAccel;
  _pitch = ALPHA * (_pitch + _gyroZ * dt) + (1.0f - ALPHA) * pitchAccel;
}
SEC 03

チルト補正コンパス（tilt-compensated compass）による yaw 推定

SEC02 で求めた roll / pitch を使って、磁力計の生値を「機体が水平だったら」の値に投影補正し、そこから yaw を求める。

3.1幾何学的な考え方

機体固定の磁力計は magX, magY, magZ という機体軸に沿った成分しか出力しない。 機体が傾いていると、この3成分の中に「本来の水平面成分」と「傾きによって混ざり込んだ鉛直成分」が入り交じっている。 そこで roll / pitch （＝機体がどれだけ・どちら向きに傾いているか）を使い、回転行列的な計算で 「機体が水平だったとしたらこう見えたはず」という水平面成分 Xh, Yh を再構成する。

FORMULA — この機体の軸配置（forward = Z, up = X）で書き直した式
TILT-COMPENSATED COMPASS — yaw
Xh = magZ*cos(pitch) - magX*sin(pitch)
Yh = magZ*sin(roll)*sin(pitch) + magY*cos(roll) + magX*sin(roll)*cos(pitch)

yaw = atan2(-Yh, Xh)   （0〜360°に正規化）
前提条件

この式が正しく機能するのは、磁力計のハードアイアン／ソフトアイアン較正が済んでいる場合のみ。 機体自体の磁性体（モーター、配線、金属フレームなど）による恒常的なオフセットや歪みを較正しないまま 傾斜補正だけを行っても、方位には系統的な誤差が残る。較正はチルト補正の「前提」であって「代わり」ではない。

IMPLEMENTATION — Arduino / C++（Sensor.cpp の変数名に合わせる）
// roll, pitch は SEC02 の相補フィルタ出力（ラジアン）を利用
// _magX/_magY/_magZ は較正済み磁力計生値
float computeYaw(float roll, float pitch) {
  float Xh = _magZ * cos(pitch) - _magX * sin(pitch);
  float Yh = _magZ * sin(roll) * sin(pitch)
            + _magY * cos(roll)
            + _magX * sin(roll) * cos(pitch);

  float yaw = atan2(-Yh, Xh);      // ラジアン, -π〜π
  float yawDeg = yaw * 180.0f / PI;
  if (yawDeg < 0) yawDeg += 360.0f;   // 0〜360°に正規化
  return yawDeg;
}

SEC 05

実装上の注意（チェックリスト）

式は正しくても、実機に載せると符号や軸の対応でつまずくことが多い。実装前に以下を確認する。

  • 軸の入れ替えで符号（chirality）が反転していないか 本ロケットは X=上・Z=前・Y=横という非標準の軸配置。基板の取り付け向きやセンサICの内部軸定義によっては、 式通りに実装しても実機では符号が逆に出ることがある。必ず実機で「傾けたら roll/pitch/yaw が期待通りの符号で動くか」を検証すること。
  • ジャイロのどの軸が roll 用／pitch 用かは配線・実装依存 本書の式では roll に gyroX、pitch に gyroZ を使っているが、これは軸配置の定義上そうなっているだけで、 実際に MPU6050 のどの物理ピン・レジスタが gyroX / gyroZ に対応しているかは Sensor.cpp 側の実装を必ず確認すること。
  • 磁気較正なしではチルト補正しても誤差が残る ハードアイアン／ソフトアイアン較正をしていない生の magX/magY/magZ に SEC03 の式を適用しても、 較正誤差に起因する系統的なオフセットは補正されない。チルト補正は「傾きによる誤差」だけを取り除くものであり、 「センサ自体・機体自体の磁気的な歪み」は別途較正が必要。
本ドキュメントの数式は指定された軸配置（X=鉛直上向き, Z=進行方向前向き, Y=横向き）に基づいて記載しています。 図中のグラフ・波形はすべて概念を示すためのイメージであり、実測データではありません。 実装前に必ず実機での軸検証・磁気較正・単体テストを行ってください。
